Token Proposal

Real OS UI HTML uses .timeline + [data-block="Navigation.TimelineItem"] + .timeline-item + .timeline-icon (wraps .timeline-icon-container + .timeline-icon-line) + .timeline-content. Background colour on the bubble comes from a shared .background-{semantic} utility (same classes used by Badge/Counter). Proposal migrates raw palette references to semantic tokens, bumps the icon to 28px so bold glyphs fit, thickens the line to 2px, and introduces a new upcoming state with a dashed line for scheduled-but-not-happened events.

Property Utility class Current (2.28) Proposed Token Rationale
icon size .timeline-icon-container 24px · glyph font-size-xs (10px) 28px · glyph 14px (phosphor-bold) — Grow to 28px. 24px with a 10px glyph reads as a pixel scribble — at small viewing sizes the ✓ inside the bubble is indistinguishable from a blob. 28px + 14px bold glyph gives enough contrast for a meaningful completion mark while keeping the bubble visually anchored to the timestamp on the left.
line width .timeline-icon-line 1px hardcoded 2px · token-border-size-050 --token-border-size-050 1px disappears on high-DPI screens and makes the timeline look broken between bubbles. 2px reads as a deliberate temporal thread and matches the Wizard connector so both step-based components share one visual language for "progression".
line colour .timeline-icon-line color-neutral-5 (#ced4da) token-border-default (#d5d5d5) --token-border-default Same visual weight, semantically anchored. Dark-mode overrides and brand-tinted themes now automatically pick up the right contrast for the connector without hitting a raw palette step.
icon bg (semantic) .background-{color} raw palette per color (e.g. color-primary #1068eb) token-semantics-{color}-base chain --token-semantics-primary-base
--token-semantics-success-base
--token-semantics-error-base
--token-semantics-warning-base
--token-semantics-info-base		 Utility classes unchanged (same HTML). Resolution chain migrates from raw palette tokens to the 2.29 semantic family — one decision point for brand / theme overrides across Badge, Counter, Timeline, and Wizard.
icon text colour .timeline-icon-container color-neutral-0 (#ffffff) — raw palette token-text-inverse (#ffffff) --token-text-inverse Value unchanged — migration from hardcoded neutral-0 to the semantic text-inverse token so any future "on-dark" override (e.g. high-contrast themes) applies automatically.
content text .timeline-right · .timeline-content-inner color-neutral-8 (#4f575e) token-text-subtle (#3b3b3b) --token-text-subtle Slightly darker subtle token improves legibility for body copy next to the bold title. Semantic anchoring means Timeline body text tracks the same scale as Card / List / Form descriptions.
content margin-bottom .timeline-content space-xl (32px) — legacy alias token-scale-800 (32px) --token-scale-800 Value unchanged — migration from the legacy --space-xl alias to the 2.29 spacing scale. Theming overrides now target one token.
upcoming state .timeline-icon.is-upcoming — (not in 2.28) dashed 2px line + outlined bubble + text-disabled body --token-border-default
--token-text-disabled		 New state. 2.28 has no concept of "scheduled but not happened" — every timeline item reads as completed history. Real product usage (audit logs, project trackers, subscription dashboards) routinely shows upcoming events alongside past ones. .is-upcoming uses a dashed line + outlined bubble so upcoming events read as "pending" at a glance without collapsing the row geometry.
timestamp label .timeline-timestamp — (unclassed in 2.28; plain body) token-text-subtlest · token-font-size-300 · semi-bold --token-text-subtlest
--token-font-size-300
--token-font-weight-semi-bold		 Formalise the timestamp slot. Real 2.28 leaves the left label unstyled, so teams reach for inline styles or body-level colour tokens — producing visually inconsistent timelines across product areas. Proposal pins it to text-subtlest + font-size-300 semi-bold so every Timeline renders the label at the same weight hierarchy.
timestamp layout .timeline-timestamp left column (fixed 88px) right-aligned via flex order: 3 (CSS-only flip) — Modern activity feeds (GitHub, Linear, Slack threads) anchor content on the left and drop the timestamp as a low-emphasis right-column — content gets the reading-order priority; the time becomes a secondary meta-field. Flipping the column is CSS-only: DOM order stays [timestamp, icon, content], proposed mode reorders visually with order. Same markup works in both layouts.
extended palette · orange .background-orange — (2.28 only has .background-warning #e9a100) token-semantics-orange-base (#ea580c) --token-semantics-orange-base New semantic colour. 2.28 ships warning (#e9a100, amber/yellow) but no true orange. Communication events (comments, notes, messaging) need a warmer tone that reads as "conversational" rather than the attention-grabbing warning. Orange #ea580c fills that gap and extends the shared .background-{color} utility used by Badge, Counter, and Timeline.
comment-card variant .timeline-content-inner.timeline-content--card — (not in 2.28) bordered card · 8px radius · subtle shadow --token-border-default
--token-border-radius-200
--token-bg-surface-default		 New variant. For rich content events (inline comments, reviewer quotes, rejection reasons) the plain inline text block doesn't scale — nested paragraphs visually merge with neighbouring rows. Wrapping the body in a bordered card (radius-200, text-default border, surface-default bg) gives it structural weight without a separate component, and the card grows with the content.
