Token Proposal

Suggested changes to improve semantic alignment and reduce hardcoded values.

Property Component Current (2.28) Proposed Token Rationale
background (track) .osui-progress-bar color-neutral-4 (#dee2e6) token-border-subtle (#f3f3f3) --token-border-subtle token-border-subtle is a lighter, less noisy track color that lets the fill stand out more clearly. The old neutral-4 value is visually heavy and not purpose-built for track backgrounds.
background (fill) .osui-progress-bar__fill color-primary (#1068eb) token-semantics-primary-base (#105cef) --token-semantics-primary-base Replacing the legacy color-primary variable with the semantic primary token ensures the fill color participates in theme overrides alongside all other primary-coloured components.
background (success fill) .osui-progress-bar--success .osui-progress-bar__fill color-success (#29823b) token-semantics-success-base (#16a34a) --token-semantics-success-base The success fill should use the new semantic success token, not the legacy palette variable, so that success state theming is consistent across all components.
border-radius .osui-progress-bar, .osui-progress-bar__fill hardcoded 100px token-border-radius-full (9999px) --token-border-radius-full Using token-border-radius-full is semantically correct for a pill shape and keeps the value in the token system. The hardcoded 100px works but is not portable across the design system.
height .osui-progress-bar (all sizes) hardcoded 4px / 8px / 12px token-scale-100 / token-scale-200 / token-scale-300 --token-scale-100 / -200 / -300 Height values that map to the scale system allow the bar to scale proportionally with density adjustments, rather than requiring three separate hard-coded overrides.
